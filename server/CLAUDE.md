## Project Overview

This is the Rust/Actix-web backend for **Hamlet**, a Discord-like chat app. It is a standalone git repo; the Tauri/Solid.js frontend lives in a sibling `client/` directory (see `../CLAUDE.md` for the combined workspace view). The server binds `127.0.0.1:3030` and the frontend expects it there.

## Commands

```bash
cargo build                                       # compile
cargo run                                         # run server on 127.0.0.1:3030
cargo test                                        # run all tests
cargo test test_message_create                    # run a single test by name
cargo test -- --nocapture                         # show println!/dbg! output during tests
RUST_LOG=debug cargo run                          # raise log verbosity (default is `info`)
```

### Code quality (run before committing)

```bash
cargo fmt                                         # format all code
cargo clippy -- -D warnings                       # lint; all warnings must be fixed
cargo audit                                       # check deps against RustSec advisory DB (requires: cargo install cargo-audit)
```

`cargo fmt` and `cargo clippy` use defaults (no `rustfmt.toml` yet). All clippy warnings must be resolved — use `-- -D warnings` to enforce this. There is no CI yet.

`Cargo.toml` enforces `unsafe_code = "forbid"`, `unwrap_used = "warn"`, and `expect_used = "warn"`. Test modules may add `#[allow(clippy::unwrap_used, clippy::expect_used)]` since `.unwrap()` is conventional in test assertions.

Rust edition 2024.

## Architecture

### Entry points
- `src/main.rs` — thin binary: connects to SQLite, creates the `Broadcaster`, seeds dev data, calls `start_server`.
- `src/lib.rs` — where everything actually lives: handlers, error types, app wiring (`configure_app`), server startup (`start_server`). This split exists so integration tests can call `configure_app` against `actix_web::test::init_service` without going through a real socket.
- `src/auth.rs` — session management: `register_user`, `authenticate_password`, `create_session`, `validate_session`, `AuthUser` extractor, cookie helpers.
- `src/middleware.rs` — `require_auth`: actix-web `from_fn` middleware that validates the session cookie and injects `AuthUser` into request extensions.

### Request path
Endpoints registered in `configure_app`. **Auth is required by default** — only register/login/logout are public. Everything else is wrapped in a scope with `require_auth` middleware.

Public:
- `POST /register`               — create account, returns session cookie
- `POST /login`                  — authenticate, returns session cookie
- `POST /logout`                 — destroy session, clears cookie (public so clients can always recover from a stale cookie)

Require auth:
- `GET  /channels`                — list all channels
- `GET  /messages/{channel_id}`   — fetch history for a channel (400s if channel missing)
- `POST /message/{channel_id}`    — insert a message, broadcast via SSE; uses authenticated `user_id`
- `GET  /messages/subscribe`      — SSE stream; returns an `actix_sse::Sse` response backed by an mpsc receiver
- `GET  /me`                      — return the authenticated user's profile

`GET /messages/subscribe` must be registered before `GET /messages/{channel_id}` in `configure_app` so actix-web's router doesn't match `/messages/subscribe` as a channel ID.

### Broadcasting (`src/broadcast.rs`)
`Broadcaster` holds `Mutex<Vec<mpsc::Sender<Event>>>`. Two constructors:
- `Broadcaster::new()` — bare instance, **no** ping loop. Tests use this so stale-client culling doesn't interfere.
- `Broadcaster::create()` — calls `new()` and spawns a 10s interval that sends a `ping` comment to every client; senders that fail to accept are dropped. This is what `main.rs` uses.

Tests can attach a fake subscriber via `Broadcaster::test_client()` (gated on `#[cfg(test)]`) which returns the raw `mpsc::Receiver<Event>` rather than an HTTP response, so you can assert on broadcast payloads without an HTTP client.

### Persistence (`src/entity/`)
SeaORM 2.0-rc entities: `user`, `channel`, `message`. `message.channel_id` → `channel.id`, `message.user_id` → `user.id`. IDs are `i64` generated by `generate_id()` (random 16-digit int, not autoincrement or UUID).

The database is **`sqlite::memory:` in `main.rs`** — the `// TODO(reno): Using in-memory for now` comment reflects the current reality. Every restart wipes data, which is why `seed_development_data` runs unconditionally on startup: it calls `get_schema_registry(...).sync(db)` to create tables, inserts a `general` channel, and registers a dev user `baipas`/`password` with a fixed session token. When swapping to a persistent DB, this seeding needs to be gated — the TODO in `main.rs` flags this.

The dev session token is printed to stdout on startup (`=== DEV: baipas session active — set cookie: session=devdev... ===`). Paste it into browser devtools to skip the login flow during development.

### Tests
Integration tests live in `tests/api.rs` with shared setup helpers in `tests/common/mod.rs`. The pattern is: `common::setup_db()` spins up a fresh `sqlite::memory:` DB with schema + a seeded channel, then each test builds the app with `test::init_service(App::new().configure(|cfg| configure_app(cfg, ...)))` and hits it with `test::TestRequest`.

One test stays in `src/lib.rs`: `test_message_create_broadcasts_to_clients`. It needs `Broadcaster::test_client()`, which is `#[cfg(test)]`-gated in the library and therefore not accessible from the separate integration test crate in `tests/`.

## Gotchas

- `Broadcaster::create` vs `new`: if a test uses `create`, the 10s ping task keeps running and can race with assertions — use `new` in tests.
- CORS in `start_server` allows **any** `http://localhost` / `http://127.0.0.1` origin with any method/header. Fine for dev, dangerous in prod (flagged with a `NOTE(reno)` in the code).
- Error responses are plain text today — `UserError` has a `TODO(reno): return errors as JSON` note; prefer to address that rather than adding a new ad-hoc error shape.
- The `.env` file is gitignored; `main.rs` does not read it, so `DATABASE_URL` there is currently unused.
